## Onboarding & Access Setup

The following access and credentials are required before running anything in the NMDC automation system. Work through these roughly in priority order — the JAWS token is only needed if you will be doing active development and testing.

**NERSC access**
- Personal user account on Perlmutter
- Access as the `nmdcda` shared account
- Recommended: configure the [sshproxy script](https://docs.nersc.gov/connect/mfa/#sshproxy) for MFA-free SSH sessions

**SPIN / Rancher**
- Required to access and manage the Scheduler container
- You may need to complete a [SPIN training](https://docs.nersc.gov/services/spin/#get-started) before access is granted

**MongoDB**
- Access to the dev and prod MongoDB tunnels via MongoDB Compass
- See the [MongoDB access guide](https://docs.google.com/document/d/11h21epiEX2HVM8pUNYPw8dzx_9p0Rx8BkpHcDk62Utc/edit?tab=t.0#heading=h.qyxkoy9mt63k) (internal)

**NMDC Runtime API**
- NMDCDA client credentials for the runtime API

**VSCode Remote Explorer** (recommended for beginners)
- Set up Remote Explorer to connect to `nmdcda@perlmutter.nersc.gov`
- Direct SSH also works: `ssh -i .ssh/nmdcda nmdcda@perlmutter.nersc.gov`

<details><summary>SSH configuration snippet</summary>

```
Host perlmutter.nersc.gov
  HostName perlmutter.nersc.gov
  User [your-username]

Host nmdcda
  HostName saul.nersc.gov
  IdentityFile ~/.ssh/nmdcda
  ForwardAgent yes
  User nmdcda
  IdentitiesOnly=yes
  ControlMaster auto
  ControlPath ~/.ssh/master-%r@%h:%p

# Use the private key generated by sshproxy.sh for NERSC servers
# Source: https://docs.nersc.gov/connect/mfa/#ssh-configuration-file-options
Host perlmutter*.nersc.gov saul*.nersc.gov dtn*.nersc.gov
    User [your-username]
    IdentityFile ~/.ssh/nersc
    IdentitiesOnly yes
    ForwardAgent yes
```
</details>

**JAWS token** (development and testing only)
- See the [JAWS configuration guide](https://jaws-docs.jgi.doe.gov/en/latest/jaws/jaws_config.html)

## Checking Workflow Status

### Using the Study Report Script

A Python script generates a workflow completion report for a given study. Run from the NERSC prod environment or from your local if you have the credentials to:

```bash
python nmdc_automation/run_process/run_report.py study-report \
    site_configuration_nersc_prod.toml \
    nmdc:sty-11-hht5sb92
```

### MongoDB Aggregation Query

The following aggregation retrieves DataGeneration records grouped by what workflow executions and jobs exist for them. It is useful for identifying gaps in the pipeline — records that are missing expected workflow executions or have jobs that may have errored. This can be used either via the [queries endpoint](https://api.microbiomedata.org/docs#/Metadata%20access/run_query_queries_run_post) on the runtime API, or in the Aggregations tab of `data_generation_set` on MongoDB Compass using just the portion in "pipeline".

<details><summary>Full aggregation pipeline</summary>

```javascript
{
  "aggregate": "data_generation_set",
  "pipeline": [
    {
      "$match": {
        "associated_studies": "nmdc:sty-11-hht5sb92",
        "analyte_category": "metagenome"
      }
    },
    {
      "$lookup": {
        "from": "data_object_set",
        "localField": "has_output",
        "foreignField": "id",
        "as": "data_object_set"
      }
    },
    {
      "$lookup": {
        "from": "workflow_execution_set",
        "localField": "id",
        "foreignField": "was_informed_by",
        "pipeline": [
          {
            "$sort": {
              "type": -1
            }
          }
        ],
        "as": "workflow_execution_set"
      }
    },
    {
      "$lookup": {
        "from": "jobs",
        "localField": "id",
        "foreignField": "config.was_informed_by",
        "pipeline": [
          {
            "$sort": {
              "config.activity.type": -1
            }
          }
        ],
        "as": "jobs"
      }
    },
    {
      "$match": {
        "$or": [
          {
            "$expr": {
              "$lt": [
                {
                  "$size": "$workflow_execution_set"
                },
                5
              ]
            }
          },
          {
            "workflow_execution_set": {
              "$not": {
                "$elemMatch": {
                  "type": "nmdc:ReadQcAnalysis"
                }
              }
            }
          },
          {
            "workflow_execution_set": {
              "$not": {
                "$elemMatch": {
                  "type": "nmdc:ReadBasedTaxonomyAnalysis"
                }
              }
            }
          },
          {
            "workflow_execution_set": {
              "$not": {
                "$elemMatch": {
                  "type": "nmdc:MetagenomeAssembly"
                }
              }
            }
          },
          {
            "workflow_execution_set": {
              "$not": {
                "$elemMatch": {
                  "type": "nmdc:MetagenomeAnnotation"
                }
              }
            }
          },
          {
            "workflow_execution_set": {
              "$not": {
                "$elemMatch": {
                  "type": "nmdc:MagsAnalysis"
                }
              }
            }
          }
        ]
      }
    },
    {
      "$match": {
        "data_object_set.in_manifest": {
          "$exists": false
        },
        "workflow_execution_set.qc_status": {
          "$exists": false
        },
        "workflow_execution_set.qc_comment": {
          "$exists": false
        },
        "has_output": {
          "$exists": true
        }
      }
    },
    {
      "$group": {
        "_id": {
          "wfex_type": "$workflow_execution_set.type",
          "job_wfex_type":
            "$jobs.config.activity.type"
        },
        "executions": {
          "$push": {
            "id": "$id",
            "wfex_id": "$workflow_execution_set.id",
            "wfex_end":
              "$workflow_execution_set.ended_at_time",
            "job_id": "$jobs.id",
            "job_wfid": "$jobs.config.activity_id",
            "job_start": "$jobs.created_at"
          }
        }
      }
    }
  ]
}
```
</details>

The query groups DataGeneration IDs by the combination of workflow executions present in `workflow_execution_set` (completed runs in good standing) and jobs in the `jobs` collection (runs that have been submitted, which may have errored if they are not also in `workflow_execution_set`). Replace `nmdc:sty-11-hht5sb92` with the study ID you are investigating.

## Diagnosing Common Issues

### Scheduler Not Creating Jobs

Check in this order:

1. **Allowlist mismatch** — Verify that the `was_informed_by` IDs in your allowlist exactly match what is in MongoDB.
   ```js
   db.jobs.find({ "config.was_informed_by": "nmdc:example-id" })
   ```
2. **Missing outputs** — Confirm that the upstream workflow execution has produced the data objects required as inputs for the child workflow.
   ```js
   db.workflow_execution_set.find({ "was_informed_by": "nmdc:example-id" })
   ```
3. **Version incompatibility** — Check that the workflow version in `workflows.yaml` is compatible with what is already in the DB (same major.minor). Use `FORCE=1` to override if needed. (not recently tested)
4. **Job already exists** — Check the `jobs` collection for an existing job for this node. The Scheduler will not create a duplicate.

### Watcher Not Picking Up Jobs

1. Confirm the `claims` array for the job is empty in MongoDB — a non-empty `claims` array means another process has already claimed it.
2. Verify JAWS / runner connectivity from the Perlmutter login node.
3. Check the Watcher logs: `watcher-dev.log` or `dev/nohup.out`.

### Job Failures

1. Inspect the `agent.state` file for the job's `last_status` and `failed_count`. 
2. Check the JAWS status:
   ```bash
   jaws status <jaws_jobid>
   ```
3. Check overall system health:
   ```bash
   jaws health --site nmdc
   ```
4. For transient download failures, try:
   ```bash
   jaws download <jaws_jobid>
   ```
5. To retry via JAWS directly:
   ```bash
   jaws resubmit <jaws_jobid>
   ```
   > **Note:** `jaws resubmit` updates the job only within JAWS. It does **not** update the Scheduler or the NMDC database. For a full pipeline resubmission, use the release endpoint described below.

### Scheduler Connection Error

The following traceback indicates the Watcher or Perlmutter has gone down, and the Scheduler can no longer reach the NMDC API:

<details><summary>Full traceback</summary>

```
Traceback (most recent call last):
  ...
  File "/src/nmdc_automation/workflow_automation/sched.py", line 316, in cycle
    wfp_nodes = load_workflow_process_nodes(self.api, self.workflows, allowlist)
  File "/src/nmdc_automation/workflow_automation/workflow_process.py", line 28, in get_required_data_objects_map
    records = api.list_from_collection("data_object_set", q)
  ...
requests.exceptions.ConnectionError: ('Connection aborted.', RemoteDisconnected('Remote end closed connection without response'))
```
</details>

**Resolution:** Check whether the Watcher process on Perlmutter is still running (see [Running the Watcher](../README.md#running-the-watcher-on-nersc-perlmutter)). If Perlmutter itself is down, monitor [NERSC system status](https://www.nersc.gov/live-status/motd/) and restart the Watcher once it recovers.
